# Промпт 28: CMS — Publish (публикация JSON)

## Контекст
Все редакторы и валидация готовы (промпты 20-27). Реализуем ключевую функцию — Publish.

## Задача
Реализуй flow публикации изменений:

1. **Кнопка Publish** (глобальная, в header CMS или отдельный раздел):
   - Показывает количество изменённых файлов (из draftManager.getChangedFiles())
   - Если нет изменений — кнопка неактивна

2. **Pre-publish flow** при нажатии Publish:
   - Финальная Zod-валидация всех изменённых файлов
   - Cross-language валидация (промпт 27)
   - Если ошибки — показать список, заблокировать publish
   - Если предупреждения (мягкий режим) — показать, запросить подтверждение
   - Показать diff: список файлов которые будут закоммичены

3. **Publish execution**:
   - Для **каждого изменённого JSON** создать **отдельный commit**:
     - commit message формат: `Update {filename} — {ISO datetime}`
   - Коммиты выполняются последовательно (не параллельно!)
   - Прогресс: «Коммит 1/5... 2/5...»
   - При ошибке: остановить, показать какие файлы закоммичены, какие нет

4. **После успешного publish**:
   - Очистить drafts для опубликованных файлов
   - Обновить SHA кеш (для будущих коммитов)
   - Показать: «Опубликовано! Deploy запущен.»

## Требования к тестам
- Pre-publish: валидация блокирует при ошибках
- Коммиты создаются по одному на файл (mock GitHub API)
- Commit message в правильном формате
- При ошибке коммита — остановка и сообщение
- После publish — drafts очищены
- Запусти тесты

## Ожидаемый результат
Полный flow публикации: валидация → подтверждение → коммиты → очистка → уведомление.
